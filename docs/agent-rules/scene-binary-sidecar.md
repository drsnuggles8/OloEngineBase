# Binary scene sidecar cache (`.scenebin`) — issue #525

Scene YAML load is dominated by yaml-cpp: for a 100k transform-only scene
(Debug), ~43 s goes to `YAML::LoadFile` tokenizing the text into a node tree and
~55 s to the per-entity node-walk (`entity["X"]`, `.as<T>()` string→number) and
`AddComponent`. Total ~98–150 s; the binary sidecar loads the same scene in ~1.5 s
— **~80–87×**. (#584 already did the cheap wins: reserve pools, dedup
`AnimatedModel`. This is the "parse once, restore fast" slice.)

The 80× win needs typed binary that skips **both** costs — skipping only the
tokenizer (a generic pre-parsed node tree) recovers <2×, because the node-walk is
the larger half. So the covered components are read/written as typed fields
straight into the component, never through a YAML node.

## What it is

`SceneSerializer::Deserialize(path)` writes a derived cache `<scene>.scenebin`
next to the source on first load and restores from it on later loads, skipping
yaml-cpp. YAML stays the source of truth; the sidecar is disposable (git-ignored
via `*.scenebin`, like the `.omesh` mesh caches). Format in
[`Scene/SceneBinaryFormat.h`](../../OloEngine/src/OloEngine/Scene/SceneBinaryFormat.h);
low-level typed I/O in [`Scene/SceneBinaryIO.h`](../../OloEngine/src/OloEngine/Scene/SceneBinaryIO.h);
container in [`Scene/SceneBinarySerializer.cpp`](../../OloEngine/src/OloEngine/Scene/SceneBinarySerializer.cpp);
per-entity component read/write (which splices the generated blocks) in
`SceneSerializer.cpp`.

## Hybrid coverage — every component, zero drift

Only ~33 of ~100 components are auto-serializable ("trivial"); the other ~67 are
hand-written in `SceneSerializer.cpp` (meshes, sprites, cameras, scripts,
particles, physics…). Rewriting all 67 in binary would duplicate the whole
serializer and carry permanent silent-data-loss risk. Instead the format is
**per-entity hybrid**:

- **kBinary** — the entity's every component is binary-covered (`IDComponent` +
  `TagComponent` + `TransformComponent` + the auto-generated trivial set). Stored
  as typed binary: fast.
- **kYaml** — the entity has any other component. Stored as the exact text
  `SceneSerializer::SerializeEntity` emits, and reloaded through the **unchanged**
  `DeserializeEntity` path. Correct for ANY component with zero reimplementation.

So a scene of thousands of simple entities (the #525 case) is all-kBinary and
gets the full speedup; a mixed scene gets a speedup proportional to its simple
fraction; nothing is ever dropped. The write always produces a sidecar — there is
no whole-scene fallback.

## Codegen, not hand-written

The binary read/write for the trivial set is generated by OloHeaderTool from the
**same `SerField` classification** the YAML `SceneSerializer` blocks use
(`EmitBinaryWriteFields`/`EmitBinaryReadFields` in `tools/OloHeaderTool/main.cpp`,
mirroring `EmitSerializeFields`/`EmitDeserializeFields`). Three generated files
(`Scene/Generated/SceneBinary{Write,Read,Covered}Components.Generated.inl`) are
`#include`d into `SceneSerializer.cpp`. Because both paths derive from one
classification, they can't drift; the `BinaryLoadReserializesIdenticallyToYamlLoad`
test pins that a binary-loaded scene re-serializes byte-identically to a
YAML-loaded one. `TransformComponent` is the one hand-written binary block (it's
non-trivial — private euler-synced rotation), kept in sync by id.

**Two independent id systems, both generated:**
- The **on-disk ComponentId** tagging each binary block is a stable FNV-1a-32 of
  the type name (`SceneBinIO::ComponentId` / `SceneBinComponentId` — keep the two
  bit-identical). Compiler-independent, so sidecars are portable. The tool
  fails the build on a collision.
- The **runtime covered-set** (per-entity "is this all-binary?" check) uses
  `entt::type_hash`, compared against `pool.info().hash()`.

## Invalidation (mirrors #454 / `binary-format-versioning.md`)

The fast path runs only when Magic matches, `Version ∈ [Min, Current]` (reject
newer — unknowable layout), the recorded `SceneSchemaVersion` equals this build's
`SceneSerializer::CurrentVersion` (a YAML-schema bump re-derives the cache rather
than migrating bytes), and the source size + last-write-time match. Decode is
**all-or-nothing**: the whole file is slurped into memory, every read is bounds-
checked, every float finiteness-checked, and any failure (short read, bad float,
unknown ComponentId) destroys the entities created so far and returns false so the
caller falls cleanly back to YAML (rewriting a fresh sidecar).

## Gotcha discovered: `SerializeEntity` during a load

The write path calls `SerializeEntity` for kYaml entities — the first time it runs
during a *load* rather than a *save*. That surfaced a latent null-deref
(`ScriptEngine::GetEntityClass` returns null with the C# engine off →
`entityClass->GetFields()` SEH-crashed). The editor save path could hit it too on
a stale/deleted script class. Now null-guarded. Lesson: any code you newly call
during load may assume save-time runtime state — audit it (the `AssetSceneLoad`
test, which loads every sandbox scene, caught this).

## Extending the covered (fast) set

The fast set grows automatically as OloHeaderTool's classifier learns more field
shapes (issue #451's ongoing work) — a newly-trivial component is picked up by
both the YAML and binary codegen with no edits here. To hand-cover a specific
complex component, add a binary block like `TransformComponent`'s (in both
`WriteEntityComponentsBinary` and the `ReadEntityComponentsBinary` switch, with a
matching `CoveredComponentIds` entry) — and bump `OSceneFormat::CurrentVersion`.
